# Solar & Battery Economic Simulation - Epic Breakdown

## Overview

This document provides the complete epic and story breakdown for the Solar & Battery Economic Simulation, decomposing the requirements from the PRD, UX Design if it exists, and Architecture requirements into implementable stories.

## Requirements Inventory

### Functional Requirements

FR1: Simulate 8,760 continuous hourly steps for a non-leap year. Terminate or truncate inputs containing 8784 hours.
FR2: Support "Do Nothing" baselines by allowing capability to bypass PV generation and/or Battery logic entirely if set to 0/None.
FR3: Support multiple PV strings with independent configurations (tilt, azimuth, capacity, PR).
FR4: Synthesize hourly consumption arrays from 12 monthly `monthly_kwh` inputs using Standard Load Profile (SLP) weighting.
FR5: Apply a strict static percentage bound (FCR_pct) splitting the battery into two virtual sub-batteries: FCR balancing vs Arbitrage.
FR6: Determine physical maximum transfer limits (Import/Export in kW) using standard Swedish 3-phase 400V grid formulas limited by the main fuse Amperes.
FR7: Determine solar curtailment volumes when panel generation exceeds both battery charge availability and the maximum grid export transfer limits.
FR8: Execute a sequential hourly mathematical loop calculating Battery State of Charge (SOC(t)) factoring in charge/discharge efficiencies dynamically based on Net load constraints.
FR9: Calculate total operational cash flows including Nord Pool Day-Ahead spot, FCR-D clearing revenues, aggregator cuts/flat-fees, VAT, and volumetric grid transfer fees.
FR10: Calculate the Skattereduktion tax credit dynamically capped at 30,000 kWh net grid transmission balancing per year.
FR11: Return a global summary metrics object with aggregated yearly values (total_money_spent_sek, total_money_earned_spot_sek, net_electricity_cost_sek, etc.).
FR12: Toggle memory expenditure dynamically via `return_timeseries` flag to either skip array preservation during Monte Carlo iterations or output a Pandas daily aggregation summary.

### NonFunctional Requirements

NFR1: Performance strict SLA: The core math simulation loop must execute all 8,760 sequential hours in under 100 milliseconds on standard CPU hardware.
NFR2: Pure Functions: Logical modules (Pv_generation, Grid_finance) must be strictly stateless functions accepting and returning arrays, with no in-place mutation.
NFR3: Data Alignment Integrity: The raw data handling layer must explicitly localize timestamps and assert that all input arrays share identical timezone alignment (`Europe/Stockholm`) to prevent data drift.
NFR4: Error Handling: Must fail-fast and throw specific ValueError exceptions for invalid bounds (e.g. FCR_pct > 1.0) prior to initiating any simulation loops.

### Additional Requirements

- Starter Template Required: Cookiecutter Data Science (CCDS) V2 must be initialized via `ccds` command. Must be scheduled as Epic 1 Story 1.
- Local Virtual Environment: Project must target local virtual environment dependency packaging (via internal Makefile), bypassing Docker.
- Apache Parquet Standardization: Intermediate data files must inherently be stored as `.parquet` to maximize IO loaded into pandas vs legacy `.csv`.
- Vector Boundary: Internal model layer exchanges must exclusively use 1D flattened `numpy.ndarray` variables (shape `(N,)`); Pandas execution loops are explicitly prohibited within `src/models/`.

## Epic 1: Project Foundation & 0-Baseline Simulation

Initialize the data-science environment and establish a functioning "Do Nothing" (0 PV, 0 Battery) baseline model capable of standardizing raw arrays to strict timelines and exporting a summary metrics object.

### Story 1.1: Initialize CCDS Data Science Architecture

As a Data Analyst,

So that I have a standardized, virtual-environment-backed folder structure isolating my raw data from my physics modules.

**Given** the project repository is empty

**Then** the standard folder structure (`data/raw`, `data/processed`, `src/solar`, `notebooks`) is generated
**And** a local virtual environment is defined in `pyproject.toml` containing numpy, pandas, and pvlib dependencies.

### Story 1.2: Standardize Raw Input Profiles to Parquet

As a Data Analyst,
I want the Data Loader script to assert timezones and export standardized Parquet arrays,
So that I am mathematically guaranteed exactly 8,760 aligned hourly steps for the simulation.

**Acceptance Criteria:**

**Given** heterogenous input data (Spot Prices, Monthly Load totals)
**When** the `src/solar/data/loader.py` script is executed
**Then** it expands monthly consumption via SLP into an 8,760-element array
**And** strictly asserts the `Europe/Stockholm` timezone across all arrays
**And** fails and truncates if leap year data (8784) is found
**And** saves the output strictly as 1D arrays into `.parquet` files.

### Story 1.3: Execute Baseline Orchestration Loop

As a Data Analyst,
I want to execute the primary simulation function with PV and Battery bypassed,
So that I can establish a control baseline representing my existing grid costs.

**Acceptance Criteria:**

**Given** the standardized Parquet input files
**When** `simulation.run_simulation()` is invoked with no solar strings and `Battery_Capacity=0`
**Then** the function skips physical equations and purely calculates "Grid Buy = Consumption"
**And** calculates financial metrics (`total_money_spent`, etc.) using the granular VAT-inclusive formula from the updated PRD 5.6 (using default 0.18 Transfer + 0.264 Tax)
**And** outputs a global metrics dictionary
**And** supports `return_timeseries=False` to bypass generating Pandas DataFrames for array preservation.

## Epic 2: Solar Array Generation Physics

Enable the analyst to calculate the raw, unrestricted hourly power generated by multiple independent solar PV arrays before any grid or battery constraints are applied.

### Story 2.1: Pvlib Array Generation Module

As a Data Analyst,
I want to covert weather irradiance arrays into a standardized Solar Power array via `pvlib`,
So that I can model the physical output of complex, multi-sided roofs.

**Acceptance Criteria:**

**Given** localized weather GHI/DNI 1D arrays and a list of configuration dictionaries representing N solar strings
**When** the `src/solar/models/pv_generation.py` function is invoked
**Then** it iterates through each string configuration to calculate Plane of Array (POA) irradiance mathematically (FR3)
**And** sums the strings into a single 8,760-element restricted `P_solar(t)` 1D Numpy array
**And** explicitly avoids mutating any external parameters during the calculation (NFR2)
**And** returns purely the `P_solar(t)` array and a subsequent `Net(t) = C(t) - P_solar(t)` array.

## Epic 3: Battery & FCR Logic

Model the sequential, hourly state-of-charge loops for a battery system, properly dedicating capacity limits across Arbitrage and Ancillary Services (FCR-D).

### Story 3.1: FCR Dynamic Capacity Bounds Allocation

As a Data Analyst,
I want the system to mathematically split the battery parameters into Virtual FCR and Virtual Arbitrage buckets prior to looping,
So that I don't simulate illegal battery market volumes or waste compute time on invalid parameters.

**Acceptance Criteria:**

**Given** the static dictionary parameter `fcr_allocation_pct` (`FCR_pct`)
**When** the battery module is initialized
**Then** it throws an immediate fail-fast `ValueError` if `FCR_pct < 0.0` or `FCR_pct > 1.0`
**And** it splits `max_power_kw` into `P_FCR` and `P_arb` constraints mathematically (FR5)
**And** it splits `capacity_kwh` into `E_arb` constraints dynamically.

### Story 3.2: Battery SOC Sequential Mathematical Loop

As a Data Analyst,
I want the battery to sequentially iterate charge/discharge dynamics over exactly 8,760 hours based on Net Energy,
So that I can accurately calculate hour-by-hour physical arbitrage limits.

**Acceptance Criteria:**

**Given** the 1D numpy array `Net(t)` (`Load(t) - Solar(t)`) and the pre-calculated boundaries `P_arb`, `E_arb`, and `eta_rt`
**When** the sequential timeline loop executes (`1 to 8760`)
**Then** it dynamically calculates instantaneous `P_charge(t)` and `P_discharge(t)` constrained strictly by available `SOC(t-1)` (FR8)
**And** it applies `eta_c` / `eta_d` explicitly during state calculations
**And** it strictly clamps the resulting `SOC(t)` between `[0, E_arb]` bounds
**And** returns purely the 1D arrays for `P_charge` and `P_discharge`.

### Story 3.3: Intraday Price Arbitrage Logic

As a Data Analyst,
I want the battery to charge from the grid during cheap hours and discharge during expensive hours,
So that I can model realistic intraday arbitrage economics beyond simple solar self-consumption.

**Acceptance Criteria:**

**Given** the 1D numpy array `spot_prices` and the pre-calculated boundaries `P_arb`, `E_arb`
**When** the sequential timeline loop executes
**Then** it identifies the $N$ cheapest hours of each day (e.g., 6 hours) to permit Grid-to-Battery charging
**And** it identifies the $M$ most expensive hours of each day (e.g., 6 hours) to prioritize discharging
**And** it maintains "Solar Priority" (excess solar always charges the battery regardless of price)
**And** it strictly obeys physical inverter power and energy capacity limits
**And** returns the updated `P_charge` and `P_discharge` vectors including arbitrage flows.

## Epic 4: Grid Fuses & Curtailment Constraints

Enforce maximum grid transmission limits based on the property's main fuse (Amperes), mathematically guaranteeing that excess solar is properly curtailed if the battery is full and the fuse is maxed.

### Story 4.1: Main Fuse Transmission Limits

As a Data Analyst,
I want the system to calculate physical main fuse transmission limits based on Ampere inputs,
So that the simulation obeys safety fuses and grid operator connection agreements.

**Acceptance Criteria:**

**Given** the `main_fuse_size_a` standard parameter
**When** the grid module initializes before simulation
**Then** it calculates `P_grid_max = (main_fuse_size_a * 400 * sqrt(3)) / 1000` specifically for 3-phase 400V grids (FR6)
**And** strictly saves it as the static maximum transmission limit constraint for the simulation.

### Story 4.2: Grid Flow & Curtailment Vector Truncation

As a Data Analyst,
I want to resolve the exact Grid Buy, Grid Sell, and Curtailment limits using fast vectorized logic,
So that I know how much power actually reached the meter.

**Acceptance Criteria:**

**Given** the complete array `Residual(t) = Net(t) + P_charge(t) - P_discharge(t)` resulting from the Battery loop, and the pre-calculated `P_grid_max`
**When** the post-loop Grid module executes
**Then** it calculates flow via parallel Numpy vector masking (avoiding Pandas/Loops)
**And** where `Residual(t) > 0`, restricts `Grid_buy(t)` to `P_grid_max` and logs the remainder as `Unmet_load(t)`
**And** where `Residual(t) < 0`, restricts `Grid_sell(t)` to `P_grid_max` and isolates the remainder mathematically as `Curtailed(t)` (FR7).

## Epic 5: Financial Metrics & Swedish Taxation

Calculate the total operational cash flow summaries by exposing Spot prices, aggregator cuts, and accurately bounding the Swedish Skattereduktion tax mechanism.

### Story 5.1: Spot Market and Aggregator Revenue Arrays

As a Data Analyst,
I want to calculate hourly net cash flows factoring in Nord Pool spot prices, taxes, and aggregator fees,
So that I can evaluate the real-world operational cash flow on an hour-by-hour basis.

**Acceptance Criteria:**

**Given** the finalized `Grid_buy(t)`, `Grid_sell(t)`, and `P_FCR` physical capacities alongside the `price_spot_hourly(t)` and `price_fcr_hourly(t)` price vectors
**When** the financial module executes
**Then** it calculates `Spend(t)`, `Earn_spot(t)`, and `Rev_FCR(t)` Numpy arrays based explicitly on the PRD pricing formulas (FR9)
**And** accurately restricts aggregator percentage fees (`aggregator_fee_pct`) solely to FCR revenue, leaving Spot revenue to the utility compensation variables
**And** correctly factors in the 25% VAT logic onto Spot purchases.

### Story 5.2: Skattereduktion Tax Rebate Dynamic Capping

As a Data Analyst,
I want to calculate the correct Skattereduktion tax credit rebate limit,
So that I model Swedish operational savings legally without accidentally over-inflating my solar returns.

**Acceptance Criteria:**

**Given** the globally summed yearly totals for Grid Buy and Grid Sell
**When** the Financial post-loop summary calculation executes
**Then** it bounds the max creditable electricity via `Max_tax_kwh = min(sum(Grid_sell), sum(Grid_buy), 30000)` ensuring no more than 30k kWh or actual-purchased kWh is credited (FR10)
**And** applies the `0.60 SEK` multiplier to calculate `Total_tax_credit_sek`
**And** compiles these summaries, subtracting the `aggregator_flat_fee_yearly`, to yield the final `net_electricity_cost_sek` scalar.
